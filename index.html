单元化 · 特点与优势

BLAST RADIUS · 爆炸半径
故障锁死在单元内

单元级隔离让任何故障的影响面 = 1/N 个分片,"全站不可用"这类事故从架构机制上被消灭,而不是靠运气规避。

SCALE-OUT · 线性扩展
容量 = 单元数 × 单元容量

用户翻倍,答案是复制一个同构单元,而不是把存量系统越堆越高 — 扩容动作标准化,不触碰已验证的架构。

FAILOVER · 切流即容灾
异地灾备 · 可用区隔离

单元跨可用区部署、异地灾备,故障转移是一次路由切换而不是一场抢修 — 7×24 交易市场里灾难恢复以分钟计。

CANARY · 真实拓扑灰度
灰度单元承接指定流量

低配单元路由少量与指定用户,新版本在真实生产架构与真实资金链路上验证 — 灰度结论可信,回切只需改路由。

DATA LOCALITY · 数据闭环
分片数据单元内自包含

交易主链路在单元内闭环,不缴"分布式事务税";跨单元的聚合与全局视图统一收口到 GZone,一致性边界清晰。

ELASTICITY · 弹性与成本
容量可度量,成本可核算

服务多实例 + 弹性缩扩容 + task 智能调度,容量按单元度量、按单元规划 — 大促加单元,平峰缩实例,成本跟着流量走。

※ 与蚂蚁 LDC、异地多活同属一个思想谱系 — 这里是它在证券交易场景下的工程化落地。

关键决策

  • 单元 = 一座完整的小交易所:行情 / 交易 / 账户 / 风控 / 出入金 / 盈亏全套微服务在单元内闭环,每个服务多实例部署,弹性缩扩容 + task 智能调度,可用区隔离 + 异地灾备
  • GZone 收口全局业务:跨单元的聚合与全局视图放在单元之上的 GZone,单元保持纯分片、互不感知 — 扩展是加单元,灾难是切单元
  • 灰度跑在真实拓扑上:专设低配灰度单元,路由少量流量与指定用户 — 新版本先在真实生产架构里烤,而不是在模拟环境里猜
  • 底座平台化:Consul 注册发现、统一缓存、日志 ES 查询、监控与全链路追踪是平台能力 — 复制单元不复制运维成本
  • 清算迁移走双轨:自研清算商对接后,百万级用户资产迁移全程新旧双轨核对,金融系统里"平滑"二字值千金
